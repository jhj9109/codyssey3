# 🧠 Mini NPU Simulator

이 프로젝트는 AI 하드웨어(NPU)의 핵심 연산인 MAC(Multiply-Accumulate)을 소프트웨어로 직접 구현하고, 데이터 크기 증가에 따른 연산 병목 현상과 NPU의 병렬 처리 필요성을 체감하기 위한 시뮬레이터입니다.

## 🚀 1. 기능 동작 검증 (Execution & Verification)

프로젝트 최상단 디렉토리에서 아래 명령어를 통해 시뮬레이터를 실행할 수 있습니다.

```bash
python main.py
```

본 시뮬레이터는 다음의 기능들을 완벽하게 수행 및 검증합니다.

- 모드 1 (사용자 수동 입력): \* 3×3 필터 2개(A, B)와 패턴을 입력받아 MAC 점수, 판정 결과, 연산 시간을 출력합니다.
  - 행/열 개수 불일치 또는 문자 입력 등 파싱 실패 시, 예외를 던지고 재입력을 유도하는 강력한 방어 로직(try-except)이 구현되어 있습니다.
  - 두 필터의 점수 차이가 허용오차(epsilon) 이내일 경우 **"UNDECIDED(판정 불가)"**로 정상 출력됩니다.
- 모드 2 (data.json 자동 평가):
  - 패턴 키(예: size*5_01)에서 문자열 파싱(split('*'))을 통해 크기 N을 안전하게 추출하고, 해당하는 크기의 필터를 정확히 매칭합니다.
  - expected 값('+', 'x')과 filter 키('cross', 'x')는 진입 단계에서 모두 내부 표준 라벨(Cross, X)로 정규화되어 일관성 있게 비교됩니다.
  - 동점(UNDECIDED) 발생 시, 기대값(Cross 또는 X)과 불일치하므로 정상적으로 FAIL 케이스로 집계됩니다.
  - 전체 테스트 완료 후 총 테스트/통과/실패 수치와 함께 실패 케이스의 원인 요약이 콘솔에 출력됩니다.
- 성능 분석: 각 모드 종료 시, 크기(N×N), 10회 평균 연산 시간(ms), 연산 횟수(N²)가 포함된 리포트 표가 출력됩니다.

## 💡 2. 코드 구조 및 설계 (Architecture & Design)

유지보수성과 테스트 용이성을 극대화하기 위해 순수 함수(Pure Function)와 컨트롤러(Controller) 패턴을 도입했습니다.

- MAC 연산 로직의 흐름 (npu_core.py):
  - 입력 패턴과 필터가 주어지면, 2중 for 루프를 사용하여 같은 인덱스 [i][j]에 위치한 요소들을 곱합니다(Multiply).
  - 곱해진 결과값을 score 변수에 지속적으로 누적하여 합산합니다(Accumulate).
  - 최종 누적된 실수형 점수를 반환합니다. (외부 라이브러리 일절 배제)
- 패턴 키 추출 및 필터 매칭 로직 (utils.py & controller.py):
  - extract*size_from_key 함수가 문자열을 * 기준으로 분할하여 두 번째 요소(숫자)를 추출합니다.
  - 이 숫자를 이용해 filters 딕셔너리에서 size\_{N} 형태의 키를 동적으로 생성하여 O(1)의 속도로 올바른 필터를 찾아 매칭합니다.
- 라벨 정규화(normalize_label)를 분리한 이유:
  - 외부 데이터(사용자 입력, JSON)의 파편화된 표현(+, cross, X 등)이 핵심 비즈니스 로직(점수 비교)까지 침투하는 것을 막기 위함입니다.
  - 정규화 로직을 분리함으로써, 핵심 연산부는 오직 표준 라벨(Cross, X)만 신경 쓰면 되므로 코드의 결합도가 낮아지고 확장성이 크게 향상됩니다.
- 시간 측정 경계 (I/O 배제):
  - 시간 측정의 정확도를 높이기 위해 파일 로드(json.load), 콘솔 출력(print) 등의 I/O 구간은 완전히 배제했습니다.
  - utils.py의 measure_mac_performance 함수 내에서 오직 mac_operation 함수가 호출되는 찰나의 구간 전후로만 time.perf_counter()를 배치하여 순수 연산 시간만 10회 반복 측정하도록 설계했습니다.

## 📊 3. 핵심 기술 원리 적용 (Core Principles)

- MAC 점수와 유사도의 상관관계:
  - MAC 연산은 두 행렬의 내적(Dot Product)과 같은 원리입니다. 필터 배열에서 패턴이 있는 부분의 값이 1(또는 양수)일 때, 입력 패턴에도 같은 위치에 1(또는 높은 양수)이 있으면 두 값을 곱해 큰 양수가 도출됩니다.
  - 이 값들이 누적되므로, 필터가 찾는 형태와 입력 데이터의 형태가 위치상으로 많이 겹칠수록(유사할수록) 최종 점수는 기하급수적으로 높아집니다.
- O(N²) 시간 복잡도의 원리:
  - 크기가 N인 2차원 배열을 모두 순회하려면 필연적으로 $N \times N$ 번의 연산이 필요합니다. 성능 분석 측정 결과, N이 3, 5, 13, 25로 커짐에 따라 연산 횟수가 9, 25, 169, 625로 제곱에 비례해 급증하며, 실행 시간 역시 이 연산 횟수에 비례하여 $O(N^2)$ 곡선을 그리는 것을 증명했습니다.
- 부동소수점 오차와 Epsilon 정책:
  - 컴퓨터는 실수를 이진수로 표현하는 IEEE 754 표준을 사용하므로, 0.1 + 0.2가 0.30000000000000004로 계산되는 등 미세한 오차가 발생합니다.
  - 이 오차로 인해 같아야 할 점수가 score_A == score_B에서 False가 되는 치명적 버그를 막기 위해, 두 값의 차이가 1e-9 보다 작으면 동점으로 간주하는 Epsilon 비교 정책을 필수로 적용했습니다.
- 동점 처리 규칙 (Mode 1 vs Mode 2):
  - Mode 1: 사용자와 상호작용하는 환경이므로, 컴퓨터가 판별할 수 없는 상황임을 사용자에게 명확히 인지시키기 위해 "판정 불가(UNDECIDED)"라는 메시지로 안내합니다.
  - Mode 2: 자동화된 테스트 환경에서는 명확한 정답(expected)과 일치하는지가 중요합니다. 동점은 기대하는 정답(Cross 또는 X)이 아니므로, 시스템적으로 '테스트 실패(FAIL)'로 규정하여 집계 로직의 무결성을 유지합니다.

### 2. 초대규모(1000×1000) 패턴 처리 시 발생하는 문제와 최적화 전략:

- 문제: 파이썬의 동적 리스트 특성상 엄청난 메모리 오버헤드가 발생하며, 단일 패턴 처리에만 100만 번의 루프 연산이 직렬로 발생하여 심각한 지연이 생깁니다.
- 최적화 적용 (보너스 과제 구현 완료): 이를 해결하기 위해 메모리상 데이터가 연속으로 배치되어 캐시 지역성(Cache Locality)을 높일 수 있는 1차원 배열(1D Array) 변환 최적화를 시뮬레이터 내부에 구현하여 2D와의 성능 비교 테스트를 완수했습니다.

### 3. CPU vs NPU 연산 병목 관점:

- CPU는 한 번에 하나의 명령어(직렬 처리)를 처리하는 데 특화되어 있어, 100만 번의 MAC 연산을 for문을 돌며 하나씩 처리해야 하는 본 문제에서 극심한 병목 현상을 겪습니다.
- 반면 NPU는 시스톨릭 어레이(Systolic Array)와 같은 구조를 통해 한 번의 클럭으로 수천~수만 개의 MAC 연산을 동시에 처리(병렬 처리)할 수 있도록 하드웨어 레벨에서 설계되었기 때문에 AI 연산에 압도적인 효율을 자랑합니다.
